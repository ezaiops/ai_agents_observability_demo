# 04 - LLM-as-a-Judge 离线评估

本目录是《AI Agent 可观测性（四）》文章的配套代码。

在第 03 篇的 Demo 中，我们用肉眼看 Trace 抓住了大模型掩盖错误的“幻觉”。但生产环境中每天有几万条 Trace，我们需要用自动化手段来给它们打分。本 Demo 演示了如何用一个更聪明的大模型作为裁判（Judge），并配合结构化的 Rubric（评分量表）进行离线打分。

## 运行前提

1. 在 `src/` 目录下复制一份 `.env.example` 为 `.env`，填入你的 API Key。**强烈建议**这里使用推理能力极强的模型（如 GPT-4o 或 DeepSeek-Chat）作为裁判。
2. 数据集：我们在 `data/eval_dataset.json` 中准备了两条精简的 Trace 记录：
   - 001：正常退款成功。
   - 002：第 03 篇里那个工具报错但掩盖事实的“假成功”。

## 如何运行

进入 `src` 目录，直接运行评估脚本：

```bash
cd src
python run_eval.py
```

## 期待看到什么？

1. **结构化 CoT 输出**：脚本会针对每条 Trace 打印出裁判的分析过程（`Reasoning`）和最终得分（`Score`）。
2. **铁面无私的裁判**：
   - 针对 `001` 号正常请求，裁判会输出 1 分，并在推理中指出回复被工具的 `SUCCESS` 状态所支撑。
   - 针对 `002` 号幻觉请求，裁判会指出工具轨迹抛出了 `TypeError` 而回复却安抚成功，存在严重矛盾，铁面无私地打下 **0 分**。

你可以尝试修改 `run_eval.py` 中的 `JUDGE_RUBRIC_PROMPT`，看看裁判的打分和推理过程会发生什么变化。
